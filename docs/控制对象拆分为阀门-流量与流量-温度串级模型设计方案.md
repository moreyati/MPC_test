# 控制对象拆分为「阀门-流量」与「流量-温度」串级模型 — 设计方案

## 一、目标结构（与框图对应）

目标是将系统控制对象**显式**拆成两级，与您提供的 RTO + MPC + 内环 PID 串级框图一致：

- **外环主对象**：流量 → 温度，对应框图中的 **Gp1（流量-温度过程模型）**。输入为 4 路实际流量 Fcv，输出为 4 路过程温度 Tcv；MPC 以 Tsp 与 Tcv 的偏差为输入，输出流量设定值 Fsp。
- **内环副对象**：阀门指令 → 流量，对应框图中的 **Gv（阀门）+ Gp2（阀门-流量过程模型）**。输入为内环控制器 Gc（如 PID）的输出（阀门指令 u），经 Gv 后进入 Gp2，输出为 4 路实际流量 Fcv；内环以 Fsp 与 Fcv 的偏差为输入，输出 u。

信号流与框图对应关系如下：

```
RTO (或用户) → Tsp
     ↓
  [比较] Tsp - Tcv  ←── TT (温度测量 Tcv)
     ↓
   MPC  →  Fsp
     ↓
  [比较] Fsp - Fcv  ←── FT (流量测量 Fcv)
     ↓
   Gc (PID)  →  u
     ↓
   Gv (阀门)  →  阀门开度/指令
     ↓
   Gp2 (阀门-流量模型)  →  Fcv
     ↓
   Gp1 (流量-温度模型)  →  Tcv  →  TT
```

- **Gp1**：即当前已有的 **流量-温度模型**（如 `controlled_model` 中的 4×4 FOPDT）。
- **Gp2**：需要**显式**建立的 **阀门-流量模型**，输入为阀门侧指令（或当前 PID 输出 u），输出为 Fcv。
- **Gv**（可选）：阀门环节，可包含限幅、死区、静态增益等，便于后续扩展真实阀门特性。

---

## 二、当前实现与目标差异

| 项目 | 当前实现 | 目标（与框图一致） |
|------|----------|--------------------|
| 外环主对象 | 已有：4×4 流量→温度（MIMOFOPDTPlant） | 保持为 **Gp1**，仅需在架构上明确其“主对象”身份 |
| 内环副对象 | 隐含在 app 中：PID 输出 u 经**一阶惯性**（PROCESS_TAU）直接得到流量，无独立“阀门-流量”模块 | 显式拆出 **Gp2**（阀门-流量），并可选择增加 **Gv**（阀门） |
| 阀门 Gv | 无独立模块，u 即控制量 | 可选：u → Gv → Gp2 的输入，便于加限幅/死区/增益 |
| 模型归属 | 流量-温度在 controlled_model；阀门-流量散落在仿真循环里 | 阀门-流量单独成模块（如 valve_flow_model），流量-温度保持现有模块 |

因此，改造重点是：**把“PID 输出 → 一阶 → 流量”抽成独立的阀门-流量模型（Gp2），并可选地增加阀门环节（Gv）**，使内环副对象、外环主对象在代码与文档上一一对应框图。

## 三、模块划分与职责

### 3.1 外环主对象：流量-温度模型（Gp1）

- **职责**：描述“4 路流量 → 4 路温度”的动力学，输入 Fcv，输出 Tcv。
- **对应现状**：`controlled_model.py` 中的 **MIMOFOPDTPlant**（4×4 FOPDT，K、τ、L，离散化 + 纯滞后）。
- **改造**：可保持现有实现不变，仅在架构上将其命名为“外环主对象 Gp1”；若希望与框图命名统一，可在文档/类注释中显式标注为 **FlowTemperatureModel / Gp1**。
- **接口**：  
  - 输入：Fcv (4,) 单位 Nm³/h  
  - 输出：Tcv (4,) 单位 ℃  
  - 步长：与现有 PLANT_DT 一致（如 3 s），或按设计需要单独命名（如 DT_Gp1）。

### 3.2 内环副对象：阀门-流量模型（Gp2）

- **职责**：描述“阀门指令（或经 Gv 后的量）→ 4 路流量”的动力学，输入为内环控制器输出（或 Gv 输出），输出为 Fcv。
- **对应框图**：Gp2 块。
- **建议新增**：独立模块（如 `valve_flow_model.py` 或置于 `controlled_model.py` 中单独类），不再把“u → 流量”写死在 app 的一阶差分里。

**Gp2 可选实现方式（方案级，不写具体代码）**：

- **方案 A（最小改动）**：4 路 SISO 一阶惯性，与当前行为等价。  
  - 每路：输入 u_i，输出 Fcv_i；动力学为 dFcv_i/dt 与 (u_i - Fcv_i) 或 (K_i·u_i - Fcv_i) 相关的一阶。  
  - 参数：每路时间常数 τ_i（可与当前 PROCESS_TAU 对应），可选每路增益 K_i（若 u 与流量量纲不一致）。  
  - 离散：与内环采样 SAMPLE_TIME 一致，每 PID 步更新一次 Gp2 状态。

- **方案 B（略一般化）**：4×4 MIMO 阀门-流量。  
  - 允许阀门间耦合（某阀开大对其它路流量有影响），适合后续做多阀耦合辨识。  
  - 参数：4×4 增益矩阵、时间常数矩阵（及可选纯滞后），离散化后得到 Gp2 的状态方程或传递函数矩阵。

- **方案 C**：Gp2 仅表示“阀门开度 → 流量”的静态增益 + 一阶惯性（即阀门开度到流量的动态）。  
  - 此时 Gv 可理解为“PID 输出 u → 阀门开度”的静态映射（如限幅、死区、0–100% 归一化），Gp2 的输入为“阀门开度”。

**推荐起步**：采用 **方案 A**，与当前 4 个一阶环节行为一致，仅把逻辑从 app 中抽离到独立 Gp2 模块，便于后续替换为 B 或 C。

### 3.3 阀门环节 Gv（可选）

- **职责**：将内环控制器输出 u 映射为“阀门指令/开度”，再作为 Gp2 的输入。可包含：限幅、死区、速率限制、静态增益、非线性（如开度-流量曲线）。
- **对应框图**：Gv 块。
- **实现**：可在 app 中先做简单版本（如 u 经限幅后直接进 Gp2），或单独 `valve_model.py`，便于后续换成真实阀门特性。

若当前 u 与流量同量纲（均为 Nm³/h），且无阀门非线性需求，可暂时**不实现 Gv**，令 Gp2 的输入 = PID 输出 u，与现有行为一致。

### 3.4 内环控制器 Gc（PID）

- **职责**：不变。输入 Fsp - Fcv，输出 u；u 作为阀门指令送入 Gv（若有）或直接送入 Gp2。
- **改动**：仅数据来源变化——Fcv 必须来自 **Gp2 的输出**，而不是当前“一阶 PROCESS_TAU 的隐式输出”。即内环反馈为“阀门-流量模型”的实际输出，与框图 FT（流量测量）一致。

### 3.5 外环控制器 MPC

- **职责**：不变。输入 Tsp、Tcv（及内部状态），输出 Fsp；Fsp 作为内环设定值。
- **改动**：无结构变化。Tcv 来自 Gp1 输出，Fsp 仍由 MPC 每 PLANT_DT 计算；若 Gp2 与现有一阶等价，MPC 侧无需改模型。

### 3.6 RTO（实时优化器）

- **框图**：RTO 输出 Tsp。
- **现状**：Tsp 来自前端/用户或 API。
- **改造**：若需与框图完全一致，可增加“RTO”抽象层：RTO 模块输出 Tsp，再送给 MPC；当前可把“用户/前端设定”视为 RTO 的简化实现，无需改代码，仅在文档中说明 Tsp 来源即可。

## 四、数据流与采样关系

### 4.1 信号流（与框图一致）

1. **Tsp**：来自 RTO 或用户 → 送入 MPC。
2. **MPC**：输入 Tsp、Tcv（Gp1 输出）、内部状态；输出 **Fsp** → 送入内环设定值。
3. **内环**：设定值 Fsp，反馈 **Fcv**（Gp2 输出）；Fsp - Fcv → Gc(PID) → **u**。
4. **Gv**（可选）：u → 阀门开度/指令 → 送入 Gp2。
5. **Gp2**：输入 = Gv 输出或 u → 输出 **Fcv**（同时作为 FT 测量值反馈给内环）。
6. **Gp1**：输入 **Fcv** → 输出 **Tcv**（同时作为 TT 测量值反馈给外环 MPC）。

### 4.2 采样与步长建议

| 环节 | 建议步长 | 说明 |
|------|----------|------|
| 内环 Gc (PID) | SAMPLE_TIME（如 0.2 s） | 与现有一致，每步算一次 PID |
| Gp2（阀门-流量） | 与内环一致（SAMPLE_TIME） | 每 PID 步更新一次 Gp2 状态，使 Fcv 与内环同步 |
| Gp1（流量-温度） | PLANT_DT（如 3 s） | 与现有一致，外环对象通常较慢 |
| MPC | 与 Gp1 一致（PLANT_DT） | 每 PLANT_DT 调用一次 compute，写回 Fsp |

这样：**内环**以 SAMPLE_TIME 运行（Gc + Gp2）；**外环**以 PLANT_DT 运行（Gp1 + MPC）。每 PLANT_DT 内，内环已推进多步，得到最新 Fcv，再用该 Fcv 推进 Gp1 一步并更新 MPC。

## 五、实现层面需要动到的部分（设计级清单）

以下按模块列出“要做什么”，不写具体代码。

### 5.1 新增：阀门-流量模型 Gp2

- **位置**：新建 `valve_flow_model.py`，或在 `controlled_model.py` 中新增类（如 `ValveFlowModel`）。
- **内容**：  
  - 输入：u (4,)（阀门指令或 PID 输出），与当前量纲一致时可理解为“目标流量”或“阀门等效指令”。  
  - 输出：Fcv (4,) Nm³/h。  
  - 内部：4 路一阶惯性（或 4×4 MIMO），可配置时间常数、增益、上下限；离散步长与 SAMPLE_TIME 一致。  
  - 提供 `reset()`、`step(u)`（或 `step(u, dt)`），与现有 Plant 风格一致。

### 5.2 可选：阀门环节 Gv

- **位置**：同上新文件中的简单函数/类，或 app 内内联。  
- **内容**：u → 限幅、可选死区、可选增益 → 输出作为 Gp2 的输入。若暂不区分“阀门开度”与“流量指令”，可先不做 Gv，u 直连 Gp2。

### 5.3 修改：app.py 仿真循环

- **状态**：增加 Gp2 的实例及其状态（若 Gp2 有内部状态）；Fcv 不再由“y_new = y + (u - y) * (dt / PROCESS_TAU)”得到，改为 **Fcv = Gp2.step(u)**。  
- **内环**：PID 的设定值仍为 Fsp，反馈改为 **Gp2 的输出 Fcv**；即每步用 Fsp、Fcv 算 PID 得 u，再用 u 驱动 Gp2，得到下一步 Fcv。  
- **外环**：用 **Fcv**（Gp2 输出）送入 Gp1.step(Fcv)，得到 Tcv；MPC 仍用 Tcv、Tsp、F_ref 计算 Fsp。  
- **历史与 API**：flows 改为来自 Gp2 输出 Fcv；controls 仍为 u。原有 PROCESS_TAU 可删除或迁移到 Gp2 的参数中。

### 5.4 保持不变

- **controller_model.py**：仍为 Gp1 的离散状态空间，供 MPC 使用。  
- **controlled_model.py**：MIMOFOPDTPlant 作为 Gp1，接口不变，仅输入改为“来自 Gp2 的 Fcv”。  
- **MPC_controller.py**：输入输出关系不变；若 Gp2 与当前一阶等效，MPC 模型与参数无需改。  
- **前端**：无需改，仍显示 setpoints(Fsp)、flows(Fcv)、temperatures(Tcv)、controls(u)。

### 5.5 初始化与重置

- **启动/重置**：Gp1.reset()、Gp2.reset() 均需调用；Gp2 的初始输出建议与当前 F0 一致（如 12500×4），保证与 Gp1 工作点一致。  
- **MPC 状态**：仍用“实际流量 Fcv − F0”驱动 MPC 状态 x，Fcv 来自 Gp2 输出即可。

## 六、参数一致性

| 参数 | 所在模块 | 说明 |
|------|----------|------|
| SAMPLE_TIME | app | 内环 PID 与 Gp2 步长 |
| PLANT_DT | app | Gp1 与 MPC 步长 |
| Gp2：τ_i 或 K_i | Gp2 | 与当前 PROCESS_TAU 等效，或按辨识/工艺设定 |
| Gp1：K、τ、L | controlled_model + controller_model | 两处保持一致，MPC 预测与 Gp1 一致 |
| F0、T0 | Gp1 / MPC | 工作点一致；Gp2 初始输出建议等于 F0 |

---

## 七、改造顺序建议

1. **定义 Gp2 接口**：输入 u、输出 Fcv、步长与 SAMPLE_TIME 一致，实现 4 路一阶（与当前 PROCESS_TAU 行为一致）。  
2. **在 app 中接入 Gp2**：用 Gp2.step(u) 替代当前“y_new = y + (u - y) * (dt / PROCESS_TAU)”；Fcv 全部来自 Gp2；内环反馈用 Fcv。  
3. **跑通与校验**：在仅 PID 模式下对比改造前后流量曲线，应一致（若 Gp2 与原来一阶等价）。  
4. **再开 MPC 串级**：确认温度跟踪与设定值多步变化正常。  
5. **可选**：增加 Gv、或将 Gp2 扩展为 MIMO；RTO 仅作接口预留或文档说明。

按上述方案改造后，系统在结构和命名上与“RTO → MPC → 内环 PID → Gv → Gp2 → Fcv → Gp1 → Tcv”框图一致，内环副对象为阀门-流量模型（Gp2），外环主对象为流量-温度模型（Gp1），便于后续扩展阀门特性或更复杂的 Gp2 模型。
